# ByteSRC（字节跳动安全响应中心）安全报告处置规则 V6.0

---

## 一、基本原则

1. 字节跳动非常重视产品及业务的安全问题，字节跳动安全响应中心（ByteDance Security Response Center）负责收取字节跳动中国区全线产品及应用的漏洞/情报。
2. 未在收录范围的非中区产品及应用的漏洞/情报，请提交至：security@bytedance.com
3. 以漏洞测试为借口，利用漏洞进行损害用户利益、影响业务运作、盗取用户数据等行为的，或以漏洞作为要挟或进行贿赂行为的，将不予奖励，并保留进一步追究法律责任的权利。
4. 未经授权任何人不得对外披露漏洞/情报细节（无论"报告"是否通过审核，包括但不限于在第三方平台上存储、传播或讨论漏洞/情报详情）或利用漏洞非法获利。如泄露了漏洞/情报信息或利用漏洞/情报非法获利，将不给予奖励、已给予的奖励有权收回、封禁平台账号。修复漏洞后公开讨论或披露漏洞，请提前联系：src@bytedance.com。
5. 字节跳动员工不得参与或通过朋友参与漏洞奖励计划。
6. 如对漏洞评级结果有争议，欢迎通过漏洞留言板或 src@bytedance.com 反馈。
7. 在 ByteSRC 平台获得奖励所产生的个税，由平台统一承担。
8. 进行礼品兑换时将收集姓名、通信地址、联系方式、身份证号码等信息用于寄送礼品和个税登记。

## 二、测试规则

- 禁止使用任何违反平台规定或法律法规的文字、图片、视频作为测试素材。
- 与模型提示和响应内容相关的问题暂不收取。
- 非信息安全漏洞导致的内容安全风险不收。
- 专项排查期间提交的漏洞/情报将不收取，排查结束后仍存在类似问题的重新正常收取。

## 三、业务系数与漏洞赏金

### 3.1 报告质量奖励

加分项（1-50分）：
- 报告结构规范、表述清晰，复现步骤详尽完整
- 主动提供可用于验证的测试账号信息
- 漏洞利用思路新颖、具有一定技术深度或创新性
- 报告有效率高
- 自评等级客观、与实际风险等级高度匹配
- 能够帮助完善防御系统，新型攻击方式、技术等提供详细分析
- 完整包含 logid 以及漏洞相关的具体接口信息

扣分/封禁等处罚项：
- 多次提交自评等级高于实际等级 2 级及以上的报告
- 多次提交无实际危害的漏洞报告
- 多次利用 AI 生成但未自行验证有效性的报告
- 提交的报告书写过于简单，无法根据报告内容复现
- 报告内容缺少漏洞触发点、漏洞描述或关键复现步骤
- 累计提交 3 个无效报告进行提醒，累计提交 5 个进行封禁

### 3.2 漏洞/情报奖励

#### 3.2.1 漏洞/情报计分规则
- 具有利用顺序关系的多个漏洞，按照多个漏洞共同影响下能达到的最大危害评分
- 同一漏洞源产生的多个漏洞，仅收取确认第 1 个为有效漏洞
- 不属于同一漏洞源但属于同一系统中同一类型/利用手段相似的漏洞，60 天内重复提交记前三个为有效漏洞
- 同一功能的越权增删改查，请合并在一个报告提交，按一个漏洞收取

#### 3.2.2 漏洞/情报系数说明
- 漏洞时效性
- 同类功能在抖店和即时零售均存在入口时，按中系数评分；如同时影响两者，按抖店高系数收一个

## 四、漏洞/情报等级评审标准

### 4.1 ByteSRC 漏洞评级指南

### 4.2 各等级漏洞详细定义

#### 1）重大漏洞
- 可获取核心业务（如电商、金融、PaaS、IaaS 等）大范围权限的漏洞
- 涉及大量核心用户敏感数据泄漏的漏洞
- 可导致核心业务拒绝服务的漏洞
- 核心业务远程命令执行

#### 2）严重漏洞
- 可直接获取服务器系统权限的漏洞（如：RCE、本地任意文件包含、反序列化等）
- 严重评级需要获取大范围权限/核心系统权限，如机器管理系统权限、重要基础设施权限
- 核心业务数据库 SQL 注入
- 存储型 XSS 漏洞、敏感信息的 JSONP 劫持
- 敏感操作的 CSRF + 蠕虫（能绑定手机号和转账）：高危
- 单台机器 RCE 并存在少量信息泄漏评级通常为【高】

#### 3）高危漏洞
- 非核心业务数据库 SQL 注入、少量敏感数据或无重要敏感数据的 SQL 注入
- 任意文件读取或操作部分非敏感文件且无法进一步利用的
- 敏感信息泄漏接口（可获取大量用户敏感信息）
- 可利用的 SSRF 漏洞
- 本地数据库注入（可造成信息泄漏或其他危害的）
- 频控缺陷漏洞，包括但不限于短信轰炸、用户账户密码暴力破解
- 支付业务逻辑漏洞利用、业务流程绕过

#### 4）中危漏洞
- 需交互的漏洞，如反射型 XSS、需要受害者点击链接、查看消息或邮件等
- 一般越权漏洞（不涉及业务核心功能或非重要敏感信息的增删改查）
- 简单 DNS 日志型 SSRF
- 轻微信息泄露漏洞（路径泄露、.git 文件泄露、Django Debug、phpinfo、CORS 敏感信息泄漏等）
- CSRF 漏洞：需明确指出利用链路、传播途径、最终造成的危害
- 通字节内部网络的无回显 SSRF，可差异化响应判断内部服务端口：中危1级
- Blind 类 SSRF：评级为【低危3级】
- 反复登出（无限登出）：中危
- 登出账号：单次登出、Self 型忽略
- 发帖、回帖、点赞：中危、低危或忽略
- 通过消耗计算、存储等资源造成短时间的拒绝服务或资源消耗攻击，评级不超过【中】
- 绕过限制使用一部分高级服务/获取优惠：中危
- 无法稳定复现的漏洞
- 一般资料修改（昵称、头像）：低危或忽略
- 利用场景复杂的漏洞（极小概率出现才能被利用并造成实质危害）
- 获取不通字节内网的测试系统的服务器权限统一评为中危1

#### 5）低危漏洞
- 轻微信息泄露
- 利用成本高、变现困难、影响范围小
- URL 跳转类需要用户登录态/需要注册固定域名来跳转的
- 可用于钓鱼或黑产的漏洞（反射型 XSS、无交互的任意 URL 跳转）
- 难以利用但存在安全隐患的漏洞（如难以利用的 SQL 注入点）
- 影响范围非常小（只有几十个注册用户、旧系统少量数据）

#### 6）忽略
- 无任何意义或影响的越权，包括无法影响他人的越权
- 无法利用或无危害的报告（恶作剧 CSRF、Self-XSS、非敏感信息泄露、无法远程利用的 dll 劫持、静态文件目录遍历等）
- 未提供成功案例，只是说明理论可行的（纯属用户猜测、无法重现、未经验证、无意义的扫描报告）
- 符合业务预期的产品设计
- 内部已知漏洞（已在网络上公开的漏洞、内部已发现、其他白帽子优先提交）
- 和安全无关的 Bug 类问题（网页打开缓慢、样式杂乱等）

#### 7）降级场景
- 利用条件苛刻的漏洞（特殊账号权限才能发现利用的）
- 需要爆破难度较大的字段（6 位及以上验证码、密码、认证信息）
- 部分环节与其他漏洞重复利用的漏洞
- 支付漏洞：如该业务存在定期核帐策略（月度核帐），评级最高不超过【中】
- 弱口令相关漏洞：以弱口令账户拥有的权限及实际危害进行评级
- 业务预期之内的资损问题、风险可控的问题
- 移动端漏洞需提供 PoC/Exploit 与验证视频及详尽漏洞分析，缺乏将影响评分
- 沙箱环境中的代码执行，如能逃逸到宿主机/物理机则正常评分
- URL 跳转类接口明确用于 URL 跳转用途无其他安全危害不收取
- 用户名遍历、SPF 邮件伪造漏洞不收取

#### 8）特殊说明
- 所有根因在 Chromium 的漏洞，需在字节自研内核启用的情况下成功利用才有效
- 如提交的漏洞不包含于上述内容中，按实际危害性、影响范围、利用难度等评估
- 如出现多个白帽子提交相同漏洞，给提交时间最早且报告详细有效的
- 白帽子主动要求忽略漏洞重新提交，需自行承担重复风险
- 关于报告"合并收取"说明

### 4.3 情报收取范围

- 用户敏感数据大规模被窃取且提供了攻击代码、漏洞利用工具等关键线索
- 服务器被入侵且提供了入侵行为特征等关键线索
- 提供字节跳动产品刷量行为的关键线索（关注、分享、点赞、评论、阅读量、播放量等）
- 提供字节跳动支付类作弊行为的关键线索（苹果 IOS 代退款等）
- 提供字节跳动游戏类作弊行为的关键线索（游戏外挂、练号打金等）
- 提供字节跳动直播类作弊行为的关键线索（刷人气、抢红包等）
- 提供字节跳动产品批量恶意账号注册、有组织薅羊毛等行为的关键线索
- 提供泄露字节跳动内部信息、用户数据等行为的关键线索（网盘、GitHub 等）
- 支付业务逻辑漏洞利用、业务流程绕过等关键线索
- 蠕虫传播、流量劫持等提供源链接、网络数据包样本等关键线索
- 核心业务数据库被下载，并提供数据库名或文件等关键线索

### 4.4 各等级情报定义

#### 1）安全情报
- 严重情报：对核心业务、系统、办公网络造成重大影响，或对集团造成大量资金损失的威胁情报。例如：超过百万级用户敏感数据泄漏事件
- 高危情报：对核心业务、系统、办公网络造成较大影响，或对集团造成较大资金损失的威胁情报。例如：有组织薅羊毛行为且传播较广
- 中危情报：对核心业务、系统、办公网络造成一定影响，或对集团造成一定资金损失的威胁情报。例如：因业务规则缺陷导致批量刷优惠券
- 低危情报：对业务、系统、办公网络造成轻微影响的威胁情报。例如：破解版移动端 APK

#### 2）业务情报
- 上报单个或少量用户的非业务规则问题导致的刷量行为
- 上报可能刷量、引流的 QQ 群号、刷量站点或黑产广告信息但未提供证据明细的
- 仅提供外部存在刷单/刷分/刷评服务，未提供证据的（必须包含可查验到的刷手真实订单、刷手账号、详细行为手法）

#### 3）无效情报
- 上报已发现或失效情报的
- 上报情报中未包含攻击路径和攻击方法说明或表述不清、逻辑不通
- 上报虚假捏造或人为制造情报信息的
- 单个电子面单截图反馈的数据泄露线索无效
- 仅提供 DNS log 的漏洞报告
- 无效情报是指错误、无意义或根据现有信息无法调查利用的威胁情报

## 六、部分漏洞类型评级补充说明

### XSS 相关漏洞
- XSS 仅能执行代码，不能获取到 Cookie 或以用户身份进行操作，评级不超过【中】
- PDF 渲染造成的相关 XSS 不收取
- 存储型 XSS 漏洞正常评级

### CSRF 漏洞
- 需明确指出利用链路、传播途径、最终造成的危害
- 敏感操作的 CSRF + 蠕虫（能绑定手机号和转账）：高危

### SSRF 相关漏洞
- 通字节内部网络的无回显 SSRF：通过差异化响应判断服务端口，评级【中危1级】
- 只能 SSRF 本机或同一网络下少量机器，造成部分信息泄露的，评级【不超过中危1级】
- Blind 类 SSRF：服务器执行攻击请求但不返回可观察响应差异，评级【低危3级】
- SSRF 统一按高系数计分
- 电商业务相关的 SSRF 漏洞将依据实际情况判定是否收取
- 电商业务无回显 SSRF：若 PSM 归属电商业务范畴则收取

### 支付漏洞
- 如该业务存在定期核帐策略（月度核帐），评级最高不超过【中】

### 频控缺陷漏洞
- 包括但不限于短信轰炸、用户账户密码暴力破解
- 短信轰炸需要绕过已有风控策略，短时间高强度（5分钟内连续发送50条以上）对同一号码轰炸
- 横向短信轰炸（对不同手机号码发送）不收取

### 电商业务漏洞
- 电商业务中的子主账号越权问题暂不收取
- 生服反爬漏洞日常仅收录严重/高危
- 同类功能在抖店和即时零售均存在入口，按中系数评分

### 大模型相关产品
- 适用产品：豆包、扣子、即梦等
- 非隐私数据泄露相关的 AI 服务风险暂不收取（jail break, prompt leak 等）
- 模型提示和响应内容相关问题举例：模型提示（越狱/安全绕过如 DAN）、模型幻觉（让大模型假装成计算机终端并执行命令）、模型响应内容（大模型输出恶意内容）
- 个人用户从 Coze 发布至豆包的智能体及插件引发的安全漏洞不收取
- 如问题对产品可造成额外的、可直接验证的安全影响（机密性、完整性、可用性），则正常收取

### 第三方组件漏洞
- 非字节（中国区）研发/运维的第三方供应商漏洞不收取
- 第三方提供维护的小程序、API 站点、服务器、营销活动、外包系统、抖音电商 ISV 等不收取
- 火山引擎、BytePlus 外部客户的漏洞不收取
- 如果提交基础组件 nday 漏洞，已公开时间在 3 个月内且字节已知晓则忽略；若字节不知晓或超过半年仍未修复，根据实际攻击效果评估
- 如果提交基础组件 0day 漏洞且有可利用证明，根据实际攻击演示效果评估

### 移动端漏洞
- 由于客户端漏洞的复杂性，需在报告中提供 PoC/Exploit 与验证视频及详尽的漏洞分析
- 客户端远程临时性拒绝服务
- 由 Auto.js、E4A、hamibot、按键精灵等框架开发的完全模拟用户点击操作的脚本工具不予收录
- 图床风险：前提是图片永不失效或可通过 ID 接口批量替换达到不失效效果

### 其他不收取场景
- Hiagent：仅限专测期间定向邀请测试，日常不收取
- 基于网络层面的拒绝服务攻击、无法影响他人的本地拒绝服务不收取
- 隐私合规相关问题暂不收取（ByteSRC 关于隐私合规类问题线索反馈问题）
- 与字节跳动（中国区）业务无关的漏洞
- 非信息安全漏洞导致的内容安全风险不收
